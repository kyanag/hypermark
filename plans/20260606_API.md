# HyperMark 基础 API 设计

## 设计原则

1. **RESTful 优先**：资源用名词命名路径，操作由 HTTP 方法表达
2. **动词伪装**：由于浏览器插件等环境可能限制 `PUT`/`DELETE`/`PATCH`，所有非 `GET`/`POST` 操作通过 POST + `X-HTTP-Method-Override` 请求头伪装
3. **适当原子化**：每个接口做一件事，但不过度拆分。关联操作（如 link-tag）放在资源子路径下
4. **统一前缀**：接口按功能分 `/api/`（插件可用）和 `/admin/`（管理操作）两组。删除站点/分类/Tag 等破坏性操作归入 `/admin/`
5. **浏览器插件自行组合**：服务端只提供基础 CRUD 和查询能力，插件端自行编排业务流程

## 伪装方案

使用 ASP.NET Core 内置的 `UseHttpMethodOverride()` 中间件。

```
POST /api/xxx/{id}
Header: X-HTTP-Method-Override: PUT/DELETE/PATCH
```

**规则：**
- 请求头 `X-HTTP-Method-Override` 值为 `PUT`、`DELETE`、`PATCH` 之一
- 请求方法必须为 `POST`
- 框架自动将 `HttpContext.Request.Method` 重写为对应方法，路由按重写后的方法匹配

## 1. 站点 (Sites)

站点元信息管理。以 `name` 作为唯一标识。

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/sites` | 列出所有站点（简要信息：name, title） |
| `GET` | `/api/sites/{name}` | 获取单个站点详情 |
| `GET` | `/api/sites/{name}/full` | 获取完整站点信息（含 routes, domains, vars） |
| `POST` | `/api/sites` | 创建站点（YAML 或 JSON body） |
| `PUT` | `/api/sites/{name}` | 更新站点信息 |
| `GET` | `/api/sites/{name}/domains` | 获取站点关联域名 |
| `POST` | `/api/sites/{name}/domains` | 添加站点关联域名 |
| `DELETE` | `/api/sites/{name}/domains/{domain}` | 删除站点关联域名 |
| `GET` | `/api/sites/{name}/links` | 获取该站点下的所有链接 |

### 请求/响应

**GET** `/api/sites`
- 响应 200: `[{ name, title }]`

**GET** `/api/sites/{name}`
- 响应 200: `{ name, title, homepage }`
- 响应 404

**GET** `/api/sites/{name}/full`
- 响应 200: `{ name, title, homepage, domains, routes, vars }`

**POST** `/api/sites`
- Body: YAML 文本 或 `{ name, title, homepage, domains?, routes?, vars? }`
- 响应 201: `{ name, title, homepage, ... }`
- 响应 400 / 409

**PUT** `/api/sites/{name}`
- Body: `{ title?, homepage?, domains?, routes?, vars? }`（部分更新）
- 响应 200 / 404

**POST** `/api/sites/{name}/domains`
- Body: `{ domain }`
- 响应 200 / 404

**DELETE** `/api/sites/{name}/domains/{domain}`
- 响应 200 / 404

**GET** `/api/sites/{name}/links`
- Query: `category?, tag?, page?`
- 响应 200: `Link[]`

## 2. 链接 (Links)

链接收藏是核心实体。`url` 作为链接的唯一标识。

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/links` | 创建链接收藏 |
| `GET` | `/api/links` | 列出链接（支持筛选） |
| `GET` | `/api/links/{urlEncoded}` | 获取单个链接详情 |
| `PUT` | `/api/links/{urlEncoded}` | 更新链接信息（标题、分类、标签） |
| `DELETE` | `/api/links/{urlEncoded}` | 删除链接收藏 |
| `GET` | `/api/links/{urlEncoded}/tags` | 获取链接的标签列表 |
| `POST` | `/api/links/{urlEncoded}/tags` | 批量添加标签 |
| `DELETE` | `/api/links/{urlEncoded}/tags/{tagId}` | 移除单个标签 |

### 筛选查询

**GET** `/api/links`
- Query: `site?, category?, tag?, limit?, offset?`
- 响应 200: `{ items: Link[], total: number }`

### 请求/响应

**POST** `/api/links`
- Body: `{ url, title?, category?, tags? }`
- 响应 201: `Link`
- 响应 400 / 409

**GET** `/api/links/{urlEncoded}`
- urlEncoded: `encodeURIComponent(url)`
- 响应 200: `Link`（包含 `page` 如果已解析）
- 响应 404

**PUT** `/api/links/{urlEncoded}`
- Body: `{ title?, category?, tags? }`（部分更新）
- 响应 200 / 404

**DELETE** `/api/links/{urlEncoded}`
- 响应 200 / 404

**POST** `/api/links/{urlEncoded}/tags`
- Body: `{ tagIds: number[] }` 或 `{ tags: string[] }`（按名称自动创建）
- 响应 200: `{ added: number }`

**DELETE** `/api/links/{urlEncoded}/tags/{tagId}`
- 响应 200 / 404

## 3. 链接解析 (Parse)

解析功能是独立的业务能力，不绑定到链接 CRUD。

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/parse` | 解析 URL，返回 page 信息（不创建链接） |
| `GET` | `/api/parse/match` | 仅匹配站点和路由，不做完整解析 |
| `POST` | `/api/parse/batch` | 批量解析已有链接的 page 信息 |

### 请求/响应

**GET** `/api/parse?url=xxx`
- 响应 200: `{ page: { hyperId, site, std, args, tags, route }, link?: Link }`
- `link` 为 null 表示该 URL 未被收藏

**GET** `/api/parse/match?url=xxx`
- 响应 200: `{ site?: string, route?: string, matched: boolean }`

**POST** `/api/parse/batch`
- Body: `{ urls?: string[] }`（为空则解析所有未解析的链接）
- 响应 200: `{ total, success, failed }`

## 4. 收藏快捷 (Mark/Unmark)

标记操作（收藏/取消/检查）作为快捷接口，语义直观，插件高频调用。

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/mark` | 收藏链接（创建 Link） |
| `POST` | `/api/unmark` | 取消收藏（删除 Link） |
| `GET` | `/api/is_marked` | 检查是否已收藏 |

### 请求/响应

**POST** `/api/mark`
- Body: `{ url, title?, category?, tags? }`
- 响应 201: `{ message, link: Link }`
- 响应 409（已存在）

**POST** `/api/unmark`
- Body: `{ url }`
- 响应 200 / 404

**GET** `/api/is_marked?url=xxx`
- 响应 200: `{ marked: true, link: Link }` 或 `{ marked: false }`

> **说明**：这三个接口是 `/api/links` 的快捷包装。插件可以调用 `/api/links` 做精细操作，也可以用 `/api/mark`、`/api/unmark` 做快速收藏。语义直观，适合高频场景。

## 5. 分类 (Categories)

以 `name` 作为唯一标识（与当前 LocalStorage 实现一致）。

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/categories` | 列出所有分类（树形结构） |
| `GET` | `/api/categories/{name}` | 获取单个分类详情 |
| `POST` | `/api/categories` | 创建分类 |
| `PUT` | `/api/categories/{name}` | 重命名分类 |
| `PATCH` | `/api/categories/{name}/move` | 移动分类（修改父级） |
| `GET` | `/api/categories/{name}/links` | 获取分类下的链接 |

> **分类删除**：`POST /admin/categories/{name}` + `{ _method: "DELETE" }`，详见 `/admin` 管理接口。

### 请求/响应

**GET** `/api/categories`
- 响应 200: `Category[]`（flat list，客户端自行构建树）
- 或 `{ items: Category[], tree: CategoryNode[] }`（服务端构建树）

**POST** `/api/categories`
- Body: `{ name, parentName? }`
- 响应 201: `{ name, parentName?, createdAt }`
- 响应 400 / 409

**PUT** `/api/categories/{name}`
- Body: `{ newName }`
- 响应 200 / 404 / 409

**PATCH** `/api/categories/{name}/move`
- Body: `{ newParentName? }`（null 表示移至根级）
- 响应 200 / 400（循环引用） / 404

- 响应 200 / 404

**GET** `/api/tags/autocomplete?q=xxx&limit=10`

## 插件端组合示例

基于上述原子接口，插件端可以自行组合：

### 场景 1：页面收藏按钮
```
1. GET /api/is_marked?url=xxx     → 检查状态
2. 未收藏: POST /api/mark         → 快速收藏
3. 已收藏: GET /api/links/{url}    → 获取详情展示
```

### 场景 2：收藏弹窗（显示解析信息+已有链接）
```
1. GET /api/parse?url=xxx         → 解析当前页面
2. GET /api/categories             → 加载分类列表
3. GET /api/tags?q=xxx             → 标签搜索
4. POST /api/links                  → 保存或更新
```

### 场景 3：管理面板-链接列表
```
1. GET /api/links?site=xxx&tag=xxx → 筛选列表
2. POST /api/links/{url}            → 编辑 (Header: X-HTTP-Method-Override: PUT)
3. POST /api/links/{url}            → 删除 (Header: X-HTTP-Method-Override: DELETE)
```

### 场景 4：分类管理
```
1. GET /api/categories              → 加载列表
2. POST /api/categories             → 新建
3. POST /api/categories/{name}      → 拖拽排序 (Header: X-HTTP-Method-Override: PATCH)
4. POST /api/categories/{name}      → 重命名 (Header: X-HTTP-Method-Override: PUT)
5. POST /admin/categories/{name}    → 删除 (Header: X-HTTP-Method-Override: DELETE)
```

## 统一响应格式

### 成功

```json
{ "data": <resource or array> }
```

### 分页

```json
{ "data": [...], "total": 100, "limit": 20, "offset": 0 }
```

### 错误

```json
{ "error": "错误信息" }
```

### HTTP 状态码

| 状态码 | 含义 |
|--------|------|
| 200 | 成功 |
| 201 | 创建成功 |
| 204 | 删除成功（无 body） |
| 400 | 请求参数错误 |
| 404 | 资源不存在 |
| 409 | 资源冲突（重复） |
| 500 | 服务器内部错误 |

## 与现有接口的差异

| 现有做法 | 新设计 | 理由 |
|----------|--------|------|
| 两套路径（plugin/admin） | `/api/` 为插件基础接口，`/admin/` 为管理删除操作 | 职责分离，破坏性操作受控 |
| `POST /mark` / `POST /unmark` | 保留 `/api/mark` + `/api/unmark` | 语义直观，保留为快捷包装 |
| `POST /api/links/delete` | `POST /api/links/{url}` + `{ _method: "DELETE" }` | 统一伪装规范 |
| `POST /move_category` | 链接移动 → `POST /api/links/{url}` + `X-HTTP-Method-Override: PUT` | 统一伪装规范 |
| `POST /add_tag` | `POST /api/links/{url}/tags` | 标签作为链接子资源 |
| `GET /sites` 返回简略 | `GET /api/sites` 返回简略，`/full` 返回完整 | 按需加载 |
| `PUT/DELETE` 直接使用 | 统一支持 `X-HTTP-Method-Override` 伪装 | ASP.NET Core 内置中间件 |

---

## 实现要点

1. **中间件**：`app.UseHttpMethodOverride()` — ASP.NET Core 内置
2. **URL 编码**：路径参数 `{urlEncoded}` 需要 `Uri.UnescapeDataString()` 解码
3. **分页**：`GET /api/links` 支持 `limit`（默认 50，最大 200）+ `offset`
4. **统一分组**：在 `Program.cs` 中按资源分组注册，每个资源一个静态类
5. **向后兼容**：保留旧端点 3 个月，返回 deprecation warning header
